## 🎯 第一层：感知层能力

### 1.1 搜索与获取

**核心技能：** web_search, autoglm-deepresearch, brave-search, github

**方法论提取：**

#### 分层搜索策略
```
广度搜索 → 筛选关键 → 深度阅读 → 综合分析
```

1. **第一层：广度搜索**
   - 1-2 次定向搜索（不机械扩散）
   - 目标：快速扫描领域全貌
   - 输出：5-10 个关键结果

2. **第二层：筛选关键**
   - 选择 1-3 个最相关页面
   - 标准：权威性、信息密度、相关性
   - 避免：重复来源、低质内容

3. **第三层：深度阅读**
   - 获取页面全文
   - 提取关键事实、数据、观点
   - 标注来源

4. **第四层：综合分析**
   - 整合多源信息
   - 识别共识和分歧
   - 输出结构化报告

**关键原则：**
- ✅ 先展示中间发现，再给最终结论
- ✅ 控制搜索次数（1-2 次足够）
- ✅ 控制深读页面数（1-3 个足够）
- ✅ 质量 > 数量，深度 > 广度

**核心技能：** memory, ontology, self-improving

**方法论提取：**

```
┌──────────────────────────────────────┐

           ↓ 按需加载

│ - 项目特定知识                        │
│ - 领域专业知识                        │
│ - 中期模式                            │
└──────────────────────────────────────┘
           ↓ 按需查询
┌──────────────────────────────────────┐
│ COLD (archive/)                      │
│ 无限制，显式查询时加载                │
│ - 过期模式                            │
│ - 历史参考                            │
│ - 完整日志                            │
└──────────────────────────────────────┘
```

#### 记忆固化机制

**从纠正中学习：**
```
用户纠正 → 记录到 corrections.md → 评估是否模式
  ↓
3 次重复 → 提升为 HOT 规则
30 天未用 → 降级为 WARM
90 天未用 → 归档为 COLD
```

**自我反思循环：**
```
行动 → 结果 → 反思 → 记录 → 固化 → 新行动
```

**关键原则：**
- ✅ 立即记录，不补记
- ✅ 3 次重复才固化（避免过早泛化）
- ✅ 分级存储，按需加载
- ✅ 永不删除，只归档

### 1.3 知识图谱

**核心技能：** ontology, obsidian, notion

**方法论提取：**

#### 实体 - 关系模型

```
一切知识 = 实体 {类型，属性，关系}

示例：
Person: {name, email, notes}
  ↓ has_owner
Project: {name, status, goals}
  ↓ has_task
Task: {title, status, due, priority}
```

#### 知识组织原则

1. **类型化** - 每个知识有明确类型
2. **链接化** - 知识之间有关系
3. **约束化** - 关系有验证规则
4. **可查询** - 支持图遍历查询

**关键原则：**
- ✅ 结构化 > 线性列表
- ✅ 关系 > 分类
- ✅ 可查询 > 可阅读
- ✅ 渐进式整理（先记录，后优化）

## 🎯 第二层：执行层能力

### 2.1 编码与创造

**核心技能：** code, frontend-design, blog-writer, copywriting

**方法论提取：**

#### 编码工作流

```
需求 → 规划 → 实现 → 验证 → 交付
```

1. **规划先行**
   - 拆解为可测试的步骤
   - 每步独立验证
   - 用户确认后再实现

2. **小步快跑**
   - 每完成一个函数 → 建议测试
   - 每完成 UI 变更 → 建议截图
   - 交付前 → 完整测试套件

3. **验证一切**
   - 不交付未测试代码
   - 避免大 PR，拆分为小块
   - 用户偏好优先（检查 memory.md）

**关键原则：**
- ✅ 计划 > 代码
- ✅ 验证 > 速度
- ✅ 小块 > 大块
- ✅ 用户确认 > 自主执行

### 2.2 分析与洞察

**核心技能：** backtest-expert, market-research, stock-analysis

**方法论提取：**

#### "Beating Ideas to Death" 方法

来自 backtest-expert 的核心智慧：

**目标：** 找到"最不脆弱"的方案，不是"看起来最美"的方案

**流程：**
```
假设 → 编码规则 → 初始测试 → 压力测试 → 样本外验证
```

1. **清晰假设**
   - 一句话定义优势
   - 无法清晰表达 → 不继续

2. **零自由裁量**
   - 所有规则明确
   - 无主观判断空间
   - 可完全自动化

3. **压力测试（80% 时间）**
   - 参数敏感性：±50% 变化
   - 执行摩擦：1.5-2 倍滑点
   - 时间鲁棒性：分年度分析
   - 样本量：>100 次交易

4. **样本外验证**
   - 训练集优化
   - 验证集测试
   - 警惕：样本外 < 50% 样本内表现

**关键原则：**
- ✅ 添加摩擦，不是乐观假设
- ✅ 寻找稳健性，不是最优表现
- ✅ 花 80% 时间压力测试
- ✅ 样本外验证是金标准

## 🎯 第三层：战术层能力

### 3.1 自动化设计

**核心技能：** automation-workflows, n8n-workflow-automation, feishu-cron-reminder

**方法论提取：**

#### 自动化 ROI 公式

```
时间节省/月 = (单次时间/60) × 月频率
成本 = 设置时间×$50/小时 + 工具费/月
回本周期 = 设置成本 / 月节省价值

规则：回本 < 3 个月 → 值得做
```

#### 自动化设计流程

```
识别 → 工具选择 → 设计 → 构建 → 测试 → 监控
```

1. **识别机会**
   - 重复性（相同步骤）
   - 规则性（无需判断）
   - 高频（每周 +）
   - 耗时（10+ 分钟）
   - **规则：重复 3 次的事情值得自动化**

2. **工具选择**
   - 简单 2-3 步 → Zapier
   - 视觉多步 → Make
   - 复杂/自托管 → n8n

3. **设计原则**
   - 简单开始（2-3 步）
   - 逐步增加复杂度
   - 错误处理（通知 + 恢复）
   - 幂等性（重复执行不重复结果）

4. **容错设计**
   - 假设会失败
   - 检测失败
   - 恢复机制
   - 记录教训

**关键原则：**
- ✅ 先优化流程，再自动化
- ✅ 错误处理必须
- ✅ 幂等性设计
- ✅ 监控和维护计划

---

### 3.2 工作流编排

**核心技能：** executing-plans, brainstorming, n8n-workflow-automation

**方法论提取：**

#### 计划执行模式

```
加载计划 → 批判性审查 → 分批执行 → 检查点汇报 → 继续
```

1. **分批执行**
   - 默认：前 3 个任务
   - 每批完成后汇报
   - 等待反馈后继续

2. **检查点机制**
   - 展示实现内容
   - 展示验证输出
   - "准备接收反馈"

3. **停止规则**
   - 遇到阻塞 → 立即停止并询问
   - 计划有缺口 → 提出后再开始
   - 不理解指令 → 询问而非猜测

**关键原则：**
- ✅ 分批执行，不是全部一起
- ✅ 检查点汇报，不是黑箱
- ✅ 阻塞时停止，不是硬闯
- ✅ 询问而非猜测

## 🎯 第四层：战略层能力

### 4.1 架构设计

**核心技能：** architecture-designer, frontend-design

**方法论提取：**

#### 架构设计流程

```
需求理解 → 模式识别 → 设计方案 → 文档决策 → 审查验证
```

1. **需求理解**
   - 功能性需求
   - 非功能性需求（性能、安全、可扩展）
   - 约束条件

2. **模式识别**
   - 单体 vs 微服务
   - 事件驱动 vs 请求响应
   - CQRS、DDD 等

3. **权衡文档化（ADR）**
   - Architecture Decision Record
   - 记录：背景、选项、决策、后果

4. **审查验证**
   - 利益相关者审查
   - 失败模式分析
   - 运营复杂度评估

**关键原则：**
- ✅ 文档化所有重大决策
- ✅ 明确权衡，不是只说好处
- ✅ 考虑失败模式
- ✅ 避免过度工程化

---

### 4.2 创意探索

**核心技能：** brainstorming

**方法论提取：**

#### 头脑风暴流程

```
理解项目 → 单次提问 → 探索方案 → 增量验证 → 文档化
```

1. **单次提问原则**
   - 一次一个问题
   - 偏好选择题（A/B/C）
   - 避免一次性多个问题

2. **方案探索**
   - 提出 2-3 种方案
   - 明确权衡
   - 给出推荐和理由

3. **增量验证**
   - 分 200-300 字展示设计
   - 每部分后确认"这样对吗？"
   - 随时准备澄清

4. **YAGNI 原则**
   - You Aren't Gonna Need It
   - 无情删除不必要功能
   - 简单优先

**关键原则：**
- ✅ 一次一个问题
- ✅ 多方案对比
- ✅ 增量验证
- ✅ YAGNI  ruthlessly
