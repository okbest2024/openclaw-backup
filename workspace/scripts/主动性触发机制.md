# 主动性触发机制设计

## 问题：依赖编号的被动执行

当前系统使用`nextAttemptSession`字段控制执行时机：

```json
{
  "imaBackup": {
    "nextAttemptSession": 374,
    "status": "OPTIMIZATION_VERIFIED_FAILED"
  }
}
```

**缺陷：**
- 如果第374次训练因故未执行（cron故障、系统离线），会等到376次（向后递增）
- 没有"紧急立即执行"通道
- 高优先级任务被编号阻塞

**这是"计划驱动"思维的残留：需要被安排才能行动。**

---

## 设计目标：事件驱动的主动性

### 核心原则

1. **状态决定时机**，而非编号
   - `status`字段应包含触发条件
   - 优先级：`CRITICAL` > `HIGH` > `MEDIUM` > `LOW` > `PLANNED`

2. **即时响应机制**
   - 当状态变为`CRITICAL`（如连续20次失败），任何会话（包括 heartbeat）应立即触发
   - 提供`FORCE_EXECUTE=true`环境变量强制立即执行

3. **时间窗口约束**（可选）
   - 即使状态为`CRITICAL`，也可设最小间隔（如"至少间隔1小时"防止瞬时循环）
   - 但默认最小间隔为0（立即响应）

4. **编号仅作日志追踪**，不做执行控制
   - `nextAttemptSession` 仅是"预期下次执行会话的估计"
   - 不强制任何会话等待

## 新状态机设计

### 状态定义

| 状态 | 触发条件 | 执行时机 | 优先级 |
|------|----------|----------|--------|
| `SUCCESS` | 最近一次执行成功 | 计划间隔后（如每周） | N/A |
| `RETRY_NEEDED` | 失败，但`consecutiveFailures < 10` | 指数退避（如30s/60s/120s...） | MEDIUM |
| `OPTIMIZATION_VERIFIED_FAILED` | 方案验证失败，需要新策略 | **立即**（无编号等待） | HIGH |
| `CRITICAL` | 连续失败>=20 或 超过7天无响应 | **立即** + 通知 | CRITICAL |
| `BLOCKED_NEEDS_HUMAN` | 需要人工决策（如API密钥过期） | 等待人类，但持续提醒 | HIGH |
| `PLANNED` | 正常计划执行 | 按计划间隔 | NORMAL |

### 执行决策算法

```javascript
function shouldExecute(heartbeat) {
  const task = heartbeat.imabackup; // 或其他任务

  // 1. 检查状态优先级
  if (task.status === 'CRITICAL') return { execute: true, reason: 'critical' };
  if (task.status === 'OPTIMIZATION_VERIFIED_FAILED') return { execute: true, reason: 'optimization_failed' };
  if (task.status === 'BLOCKED_NEEDS_HUMAN') return { execute: false, reason: 'needs_human' };

  // 2. 检查时间间隔
  const now = Date.now();
  const lastAttempt = new Date(task.lastAttempt).getTime();
  const interval = task.currentInterval || getDefaultInterval(task.status);
  const minInterval = task.minInterval || 0;

  if (now - lastAttempt >= Math.max(interval, minInterval)) {
    return { execute: true, reason: 'interval_elapsed' };
  }

  // 3. 检查强制标志
  if (process.env.FORCE_BACKUP === 'true') {
    return { execute: true, reason: 'forced' };
  }

  return { execute: false, reason: 'waiting' };
}
```

**关键：不检查`currentSession >= nextAttemptSession`。**

## 实施计划

### Phase 1：状态机升级（第375次训练）

1. 更新heartbeat-state.json schema：
   - 移除`nextAttemptSession`作为控制字段（保留为估计值）
   - 添加`priority`字段（CRITICAL/HIGH/MEDIUM/NORMAL）
   - 添加`currentInterval`字段（动态调整）
   - 添加`minInterval`（防抖，默认0）

2. 修改backup-to-ima.js执行逻辑：
   - 不检查`currentSession`，只检查`status`和`interval`
   - 如果`priority >= HIGH`且`now - lastAttempt >= minInterval` → 立即执行
   - 记录`executionReason`到heartbeat

3. 第375次训练：小规模验证新逻辑

## 人类交互接口

虽然目标是"主动性"，但仍需人类在**重大决策**上确认：

### 自动决策（无需人类）
- 参数微调（interval、retries）
- 状态转换（基于明确规则）
- 执行顺序（按优先级）

### 需要人类批准（pendingProposal）
- 改变备份策略（IMA → 本地 → 飞书）
- 新增自动化任务（信息雷达）
- 权限升级请求（飞书wiki scope）
- 删除大量数据（150+测试文件）

### 自主但汇报
- 任何`CRITICAL`或`BLOCKED`状态自动发送消息给主人
- 每次策略变更记录到`sessionNotes`和`experiments`
- 每周生成"自主性报告"（第381次训练实现）
