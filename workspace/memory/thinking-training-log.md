### 思维过程

#### 1. 系统思维：识别系统边界、组件与反馈结构

**系统定义：** OpenClaw 备份系统（backup-to-doc-table.js）
- 边界：从文件读取 → 哈希计算 → 表格写入 → 状态记录
- 核心组件：
  1. **源文件监控器**：扫描 workspace 中需备份的文件列表
  2. **状态管理器**：读取/写入 memory/feishu-backup-state.json（缓存哈希、记录统计）
  3. **文档接口**：调用 feishu_doc API（create_table, write_table_cells, list_blocks）
  4. **错误处理器**：捕获 API 失败、权限错误、网络异常
  5. **触发器**：cron 每4小时执行一次

**当前架构（单点失效）：**

```
[Workspace Files]
       │
       ▼
[backup-to-doc-table.js] ──→ [feishu_doc API]
       │                            │
       └─────→ [state.json] ←──────┘
```

**单点风险：**
- 如果文档被意外删除 → 所有备份丢失（无副本）
- 如果 feishu_doc API 持续失败（rate limit 或 权限变更）→ 备份停滞
- 如果 state.json 损坏 → 无法追踪变更，可能重复上传或遗漏
- 如果脚本本身错误 → 无防护

**系统思维建模：想要什么？一个**抗失效的备份系统**，具备以下特性：**
- **冗余**：关键数据有多个副本（不同存储介质/位置）
- **隔离**：错误不会级联（例如，一个文件备份失败不影响其他文件）
- **可检测**：能及时发现数据丢失或损坏
- **可恢复**：能回退到已知良好状态
- **自愈**：自动修复或重建损坏部分

**反馈回路分析：**

**回路1（正常循环回路 - 负反馈稳定）：**
```
[检测变更] → [计算哈希] → [对比历史] → [确定需备份文件] → [上传表格] → [更新状态] → (闭环)
```
- 这是主要操作回路，稳定运行

**回路2（错误累积回路 - 正反馈危险）：**
```
[API失败] → [记录错误但不处理] → [下次继续尝试] → [错误日志堆积] → [state.json膨胀] → [性能下降] → [更多失败]
```
- 正反馈：失败导致更多失败，需要打断

**回路3（数据丢失回路 - 正反馈灾难）：**
```
[文档意外删除] → [无感知] → [下次备份覆盖写入] → [历史数据永久丢失] → [恢复能力缺失]
```
- 这是最危险的回路，必须有检测机制

**回路4（状态依赖回路 - 脆弱性）：**
```
[state.json 损坏] → [无法读取历史哈希] → [所有文件视为"需要备份"] → [API 调用激增] → [触发 rate limit] → [全部失败]
```
- 单点依赖导致级联失败

**杠杆点（Meadows 12个杠杆）：**

1. **参数：state.json 的结构设计**（高杠杆）
   - 当前：扁平键值对 {filePath: {hash, lastBackupTime, size}}
   - 升级：分段存储（daily snapshots），保留最近7天的完整状态快照
   - 杠杆：使状态文件本身可恢复，避免单点损坏

2. **信息流：备份前的完整性验证**（高杠杆）
   - 当前：上传后不验证内容是否正确
   - 升级：上传后立即读取表格数据，验证哈希匹配
   - 杠杆：早期检测 API 静默失败

3. **规则：失败重试策略**（中杠杆）
   - 当前：失败后记录，下次继续
   - 升级：指数退避 + 错误分类（可恢复 vs 不可恢复）
   - 可恢复（网络抖动）→ 重试3次
   - 不可恢复（权限错误）→ 警报并跳过，不阻塞其他文件

4. **自组织：文件分组隔离**（中杠杆）
   - 当前：所有文件在同一个流程，一个错误可能影响整体
   - 升级：按文件类型/目录分组（config, memory, scripts, skills）
   - 分组独立失败，不影响其他组
   - 杠杆：局部化错误，防止级联

5. **目标：引入外部冗余**（高杠杆）
   - 当前：仅存于飞书云文档
   - 升级：增加第二副本（本地压缩包 + 另一云存储如 GitHub/Gitee）
   - 3-2-1 原则：3份副本，2种介质，1份异地
   - 杠杆：从根本上降低丢失概率

6. **系统结构：分离备份与上传**（高杠杆）
   - 当前：边读边上传，无中间产物
   - 升级：先创建本地备份包（tar.gz），再上传
   - 本地包也作为第二份副本
   - 杠杆：本地可作为恢复源，即使云端全丢也有保底

7. **认知：错误日志的自动分析**（低杠杆但预防）
   - 记录失败模式，如果同一错误连续出现3次，升级告警级别
   - 防止"麻木"：不忽视重复错误

8. **范式：预防 > 检测 > 自愈 > 回退**（最高杠杆）
   - 这是可靠性工程的黄金层次
   - 当前：只有预防（定期备份）
   - 需要补全：检测、自愈、回退

#### 2. 批判性思维：证据评估与偏见识别

**评估当前方案的证据质量：**

**主张：** "云文档表格方案已全面验证，适合生产"
**证据链：**
- 测试1：create_table 成功 ✅
- 测试2：write_table_cells 成功 ✅
- 测试3：list_blocks 成功 ✅
- 结论：API 稳定，可用

**批判分析：**

1. **样本偏差（Sample Bias）**：
   - 测试环境 vs 生产环境：测试用单个文档，小数据量（几行）；生产将写入数千行，性能未验证
   - 时间偏差：测试是"一次性成功"，未测试长时间运行（30天后 API 行为是否变化）

2. **确认偏误（Confirmation Bias）**：
   - 我记住了 successful tests，但忽略了可能的 edge cases：
     - 表格行数上限？未测试到 1000+ 行
     - 并发写入？未测试多个 cron 同时触发
     - 部分更新？只测试全表覆盖，未测试"只追加"的性能
   - 需要**主动寻找反例**：尝试写入 5000 行，看是否失败

3. **因果推理错误（Post Hoc）**：
   - "因为测试成功，所以生产没问题" → 未考虑规模变化带来的新问题
   - 相关性 ≠ 因果：API 在小数据量成功，不保证大数据量成功

4. **轶事证据（Anecdotal） vs 系统证据**：
   - 当前证据是"我测了几个操作" → 轶事
   - 需要：系统化压力测试 + 监控 + SLO（服务水平目标）

5. **可用性 vs 可靠性混淆**：
   - 批判：**"可用" ≠ "可靠"**
   - 可用：API 能响应
   - 可靠性：长期运行不出错，数据不丢，可恢复
   - 当前方案缺少：**数据一致性检查、版本历史、删除防护**

**识别认知偏见：**

- **过度自信偏见**：基于几次成功测试就认为方案成熟
- **现状偏见**：因为已经部署，就倾向于认为它足够好，不愿投入额外改进
- **沉没成本**：已在云文档方案投入（时间、配置），不愿意承认需要第二副本
- **可得性启发**：因为没发生数据丢失，所以认为丢失概率低（实际未知）

**批判性结论：** 当前方案**可用但脆弱**。需要主动寻找弱点，而不是等待失败。三阶段升级不是"锦上添花"，是**纠正认知偏见**的必须步骤。

#### 3. 创造性思维：生成新奇解决方案与概念合成

**脑暴候补方案：**

A. **本地 Git 仓库备份**（传统但有效）
   - 每次备份前 git commit，推送到 GitHub/Gitee
   - 优点：有完整版本历史，分支，回退容易
   - 缺点：需要 git 知识，二进制文件（如图片）效率低

B. **加密压缩包 + 多位置存储**（简单粗暴）
   - tar.gz 所有文件，加密（openssl），上传到：
     1. 飞书云文档（primary）
     2. GitHub Gist（secondary）
     3. 本地磁盘（tertiary）
   - 优点：各存储独立，单一失效不影响
   - 缺点：无法单独恢复一个文件，必须解压整个包

C. **事件溯源（Event Sourcing）**（高级）
   - 不存储文件内容，只存储变更事件（文件修改记录）
   - 可以从初始状态 + 所有事件重建任意时刻的文件
   - 优点：极省空间，有完整历史
   - 缺点：实现复杂，恢复速度慢

D. **分层存储（Hot/Warm/Cold）**（平衡）
   - Hot: 云文档表格（最新24h，快速查询）
   - Warm: 本地 daily tarball（7天滚动）
   - Cold: 云端冷存储（如 AWS Glacier / 异地服务器，30天+）
   - 优点：成本/性能平衡，多重保护
   - 缺点：管理复杂度高

**概念合成：融合最佳元素**

**方案 E（推荐）：混合可靠性体系**

```
┌─────────────────┐
│   主备份：飞书  │ ← 快速查询，人眼可读
│   云文档表格     │
└────────┬────────┘
         │ 成功写入后
         ▼
┌─────────────────┐
│  次要备份：本地 │ ← 快速本地恢复，不依赖网络
│  tarball (7天)  │    tar.gz + 增量 (rsync)
└────────┬────────┘
         │ 完成后
         ▼
┌─────────────────┐
│  远程副本：Git  │ ← 版本历史，异地容灾
│  (或 Gitee)     │
└─────────────────┘
```

**同步机制：**
```
backup-to-doc-table.js
       │
       ├─▶ 步骤1：打包本地 tarball (增量)
       │
       ├─▶ 步骤2：上传到云文档（主）
       │        └─ 验证哈希匹配
       │
       └─▶ 步骤3：git commit + push（远程）
```

**创新点（概念合成）：**
1. **表格 + 压缩包**：表格提供人眼可读、可查询；压缩包提供完整恢复能力
2. **异步复制**：步骤1、2、3可以是异步的，不阻塞主流程（步骤2成功后步骤3可稍后执行）
3. **状态快照化**：state.json 不再是一个文件，而是 daily rotating snapshots：
   - `state-2026-03-28.json` (today)
   - `state-2026-03-27.json` (yesterday)
   - `state-latest.json` (symlink)
   - State 文件本身也进入备份循环

4. **删除防护**：
   - 云文档：启用回收站保留策略（30天）
   - Git：永远不删除 commit，即使文件被移除
   - 本地：使用 immutable storage（chattr +i 防删，或备份到外部硬盘）

5. **检测机制**：
   - 每次备份后，检查：
     - 表格行数是否合理（例如，如果只剩1行，说明可能被清空）
     - 文件哈希是否匹配（通过后台读取验证）
     - state.json 的 schema 是否正常
   - 如果检测到异常：
     - 触发紧急警报（cron 发送消息）
     - 自动从本地 tarball 恢复表格
     - 记录 incident 到 `memory/backup-incidents.md`

**自愈设计：**

**场景：云文档被意外删除**
- 检测：list_blocks 返回 empty 或 expected table block 不存在
- 决策：如果本地 tarball 存在且是最新的（<24h），自动 recreate 表格并重写
- 日志：记录"自愈事件"到 incidents 文件，并发送通知

**场景：state.json 损坏（JSON parse error）**
- 检测：try-catch 解析失败
- 决策：加载最近一天的 snapshot (state-YYYY-MM-DD.json)
- 自愈：用 snapshot 重建 state-latest.json，继续运行

**场景：Git push 失败（认证过期）**
- 检测：git command exit code != 0
- 决策：记录到 `git-failures.log`，不阻塞主流程
- 后续：手动修复凭证，重新 push（或下次自动重试）

**回退策略：**
- 如果新方案（三副本）本身引入严重 bug，如何回退？
- 保持旧版脚本备份：`backup-to-doc-table.js.v1`（单副本版）
- 回退步骤：cron 改回调用 v1，数据不丢失（因为三副本包含旧数据）

### 结论与洞察

#### 核心结论：三层次可靠性体系

**层次1：预防（当前已部分具备）**
- 定期备份（cron 4h）
- 权限隔离（只给 docx:document:write）
- 完成度：70%

**层次2：检测（需新增）**
- 备份后立即验证（读取表格确认写入成功）
- 表格完整性检查（行数、关键字段存在）
- state.json 快照化（daily rotation）
- 完成度：0% → 需实现

**层次3：自愈（需新增）**
- 文档删除检测 → 自动从本地 tarball 恢复
- state.json 损坏 → 自动加载最近快照
- API 失败 → 分类重试 + 隔离
- 完成度：0% → 需实现

**层次4：回退（需新增）**
- 版本化脚本（保留 v1、v2）
- 本地 tarball 作为恢复源
- 远程 git 副本作为终极保险
- 完成度：0% → 需实现

**渐进式实施路线图：**

| 阶段 | 工作量 | 新增组件 | 风险覆盖率 |
|------|--------|----------|-----------|
| 阶段0（当前） | 0 | 单表格 + 单 state.json | 30% |
| 阶段1（1-2天） | 4h | + 验证 + 快照 + 删除检测 | → 60% |
| 阶段2（3-4天） | 8h | + 本地 tarball + 自愈逻辑 | → 85% |
| 阶段3（1周） | 12h | + git 远程副本 + 完整监控告警 | → 95% |

#### 具体行动项（阶段1）

1. **状态快照化**（1小时）
   - 修改 backup-to-doc-table.js，每次运行时：
     - 如果 `state-latest.json` 存在，复制到 `state-backup/state-YYYY-MM-DD-HH.json`
     - 保留最近 48 个快照（2天）
   - 代码位置：`backupState()` 函数末尾

2. **备份验证**（1小时）
   - 写表格后，增加 `verifyBackup()` 函数：
     - 调用 `feishu_doc list_blocks` 读取刚写入的表格
     - 检查是否包含刚备份的所有文件名
     - 如果缺失，记录错误并尝试重写（最多1次）
   - 失败超过2次，发送警报

3. **删除检测 + 自愈**（2小时）
   - 在脚本开始时，先检查目标表格是否存在
   - 如果不存在：
     - 从 `state-backup/` 加载最新快照，重建内存状态
     - 调用 `create_table` 重新创建表格
     - 读取所有需要备份的文件，全量重写
     - 记录"自愈事件"到 `memory/backup-incidents.md`
   - 注意：自愈前需发送警报，人工确认（避免误操作覆盖新数据）

#### 可复用思维模式

**模式85：可靠性工程的四层次防御**
- 预防：减少失败概率（定期备份、权限控制）
- 检测：及时发现失败（验证、监控、完整性检查）
- 自愈：自动修复（检测到删除后重建）
- 回退：安全降级（保留旧版本、多副本）
- 应用：任何关键系统（数据库备份、配置文件管理、证书轮转）

**模式86：系统思维的杠杆点优先排序（Meadows 应用）**
- 识别所有可能的干预点
- 分类：参数（易改）、信息流（中）、规则（难）、目标（极难但高杠杆）
- 选择：优先高杠杆且易改的（如参数 + 信息流）
- 避免：过早改变目标或 Paradigm（太贵，风险大）

**模式87：批判性思维的证据质量检查清单**
- 样本偏差：测试数据是否代表生产？
- 确认偏误：是否只看了成功案例？
- 规模效应：小数据成功是否意味着大数据成功？
- 时间效应：一次性成功是否意味着长期稳定？
- 使用此清单，在"方案可用"结论前必须全部回答

**模式88：创造性概念的合成法则**
- 不追求全新，而是**杂交**：A方案的优点 + B方案的优点
- 本例：表格（可读性） + 压缩包（完整性） + Git（版本历史） = 混合方案
- 公式：`最佳方案 = 现有元素的新排列组合，而非从零发明`

**模式89：最小可行可靠性升级（MV Reliability）**
- 类似 MVP，但针对可靠性：用最小成本覆盖最常见风险
- 方法：风险矩阵（概率 × 影响）→ 优先处理高概率/高影响
- 本例：删除风险（中概率，灾难性影响）→ 阶段1必须覆盖
- 避免："完美可靠"目标，抵制范围蔓延
