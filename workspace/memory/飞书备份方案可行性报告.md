## 🎯 推荐备份策略

### 架构设计

```
┌────────────────────────────────────────────────────┐
│                 飞书云文档表格                      │
│          (元数据备份 - 主存储)                      │
│  • 文件名、路径、哈希、时间、大小、状态、备注        │
│  • 文档: OpenClaw 备份索引                         │
│  • 表格: 8列（设计见下）                           │
└───────────────────────┬────────────────────────────┘
                        │
                        │ 自动同步
                        ↓
┌────────────────────────────────────────────────────┐
│          本地文件 + Git 备份（辅助）                │
│  • workspace 源文件                                │
│  • Git 提交记录 + GitHub 远程仓库                   │
└────────────────────────────────────────────────────┘
```

**说明**：
- **元数据**（文件清单、哈希、状态）存储在飞书云文档表格，便于查询、筛选、管理
- **文件内容**继续使用本地文件系统 + Git 备份（已有机制）
- **Drive API 暂不使用**（当前故障，且非必需）

### 表格字段设计（8列）

| 列名 | 类型 | 说明 | 示例 |
|------|------|------|------|
| 文件名 | Text | 文件名（用于显示） | `heartbeat-state.json` |
| 文件类型 | Text/SingleSelect | 分类：配置文件/记忆文件/日志文件/脚本/技能/其他 | `配置文件` |
| 本地路径 | Text | workspace 相对路径 | `memory/heartbeat-state.json` |
| 备份时间 | Text | ISO 8601 格式 | `2026-03-25T01:52:00Z` |
| 文件大小 | Number | bytes | `2048` |
| 文件哈希 | Text | SHA256 截断（用于去重） | `sha256:abc123...` |
| 状态 | Text/SingleSelect | 成功/失败/跳过 | `成功` |
| 备注 | Text | 错误信息或说明 | `文件首次备份` |

### 脚本逻辑

1. **扫描**：遍历 workspace，匹配 `includeGlobs`，排除 `excludePatterns`
2. **计算**：对每个文件计算 SHA256 哈希、获取大小、修改时间
3. **去重**：读取表格现有记录 → 建立 `{path: {hash, time}}` 缓存 → 对比哈希
4. **准备**：只收集新增/变更的文件，构建 2D 数组 `values[row][col]`
5. **追加**：使用 `write_table_cells` 写入表格末尾（row_start = 当前行数）
6. **记录**：更新本地状态文件 `memory/feishu-backup-state.json`
7. **错误处理**：捕获异常，记录到"状态"和"备注"列

### 执行频率

- **heartbeat 检查**：每 4 小时（白天）
- **触发条件**：检查备份状态文件，如果上次备份超过 4 小时 → 执行
- **dry-run 选项**：测试时使用 `--dry-run` 参数不实际写入

## 💡 核心发现与洞察

### 为什么云文档表格是更优解？

1. **技术债务为零**：API 设计合理，无已知 bug（Bitable 的 DateTime 字段问题暴露了其内部不稳定性）
2. **规模无上限**：文档大小限制 vs Bitable 的 10k 行限制，适合长期累积
3. **权限最小化**：仅需文档权限，无需 tenant 级 bitable 权限
4. **用户体验**：在飞书中直接查看、筛选、排序、导出 CSV 都很方便
5. **维护成本**：单文档管理，无需单独维护 Bitable 应用结构

### 研究过程中的关键决策

- **放弃 IMA 备份**：rate limit 无法绕过，C2 策略（每周一次）只是推迟问题，不如彻底迁移
- **Bitable vs 云文档**：虽然 Bitable 基础 CRUD 可用，但云文档表格的简单性和无限制性使其成为更好的长期选择
- **Drive 文件存储**：当前 AP I异常，且元数据备份已足够，暂不投入精力修复

### "做实验" vs "理论分析"

本次研究中，**实践验证** 起了决定性作用：
- 理论上 Bitable 可能是最佳选择（字段类型丰富）
- 但实践发现 DateTime 字段 bug，且云文档表格更简单
- **没有亲手测试 create_table + write_table_cells，就会错过最佳方案**
