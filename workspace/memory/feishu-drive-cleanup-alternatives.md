# Feishu Drive 清理替代方案

**问题：** Drive API 返回 400 错误，所有操作失败（drive:file:create 可能缺失或 endpoint 错误）

**现状：**
- 测试/备份文件数量：156 个
- 文件位置：飞书云盘根目录及子目录
- 影响：仅占用空间，不影响核心功能

**替代方案评估：**

### 方案1：手动网页清理（推荐立即执行）
- **方法**：登录飞书网页版 → 云盘 → 搜索"测试"/"backup"关键词 → 批量删除
- **时间**：约 10-15 分钟（手动操作）
- **风险**：低，需谨慎确认文件列表
- **成本**：人类干预（主人操作）

### 方案2：等待 Drive API 修复（长期）
- **原因**：当前权限可能缺少 `drive:file:create` scope，或者 endpoint 错误
- **行动**：请求 feishu 管理员升级权限，或调查 API 文档
- **风险**：修复时间不确定，可能根本不可用（某些 Drive 操作可能受限）
- **成本**：等待 + 调试时间

### 方案3：使用 feishu_wiki API 替代（如果权限允许）
- **前提**：需要 wiki 的 tenant 级别权限（当前为 user 类型）
- **评估**：目前 wiki spaces 返回 400，同样权限问题，不可行
- **结论**：暂不考虑

### 方案4：忽略，等待定期归档（被动）
- 既然不影响核心功能，可暂不处理
- 定期（如每月）手动清理一次
- 未来如果 Drive API 修复，再自动化

### 方案5：本地 markdown 文件 + Git 替代云盘（战略转变）
- 将备份从云盘转向本地 Git 仓库
- 利用 GitHub/GitLab 作为远程备份
- 无需任何飞书 API，完全自主控制
- 代价：失去飞书内的可视化查看（但表格索引已提供元数据）

**推荐行动组合：**

1. **短期（当前）**：
   - 执行方案1：主人在方便时手动清理 156 个测试文件
   - 清理标准：所有包含 "test"、"backup"、"debug" 的文件；所有 scripts/ 下的临时文件
   - 保留：核心配置文件、技能文件、真实记忆文件

2. **中期（本周）**：
   - 调研 Drive API 权限问题（找 feishu 管理员或查文档）
   - 如果无法修复，正式转向 Git 备份策略（备份元数据到云文档表格，文件内容到 Git）

3. **长期（架构）**：
   - backup-to-doc-table.js 完成 → 元数据 backup 到云表格
   - 增加 git commit 逻辑 → 文件内容 backup 到 Git
   - 完全弃用 IMA 和 Drive，实现自主备份体系
