- ❌ **缺乏模型降级机制**：主模型失败时，没有自动切换到备用模型
- ❌ **任务依赖外部服务可用性**：反思任务完全依赖 LLM API，无离线备选方案

**学到了什么（具体、可操作的教训）：**

1. **实现模型故障转移机制**：
   - 创建模型优先级列表（至少 2 个不同提供商）
   - 在配置中添加 `fallback_models` 字段
   - 主模型失败时自动尝试降级模型，记录切换日志

2. **添加健康检查预执行**：
   - 在关键 cron 任务开始时添加 `providerHealthCheck()`
   - 检查所有依赖服务的可用性和配额
   - 健康检查失败时发送告警并跳过任务（避免无效执行）

3. **建立服务状态监控面板**：
   - 在 heartbeat-state.json 中添加 `providerStatus` 字段
   - 记录各提供商的信用余额、API 状态、最后检查时间
   - 在 HEARTBEAT.md 中加入提供商健康检查（每 4-8 小时）

4. **改进错误处理策略**：
   - 区分错误类型：credits exhausted、rate limit、network、server error
   - 针对 credits exhausted：发送告警消息给主人，标记为"需要人工干预"
   - 针对临时错误（rate limit、network）：自动重试（指数退避）
   - 针对永久错误（invalid key）：记录并停止尝试

5. **任务执行状态追踪**：
   - 在 `cron/runs/{jobId}.jsonl` 中确保记录成功/失败状态
   - 失败时记录错误类型、错误信息、时间戳
   - 积累失败次数，超过阈值触发升级告警

## 🧠 今日核心洞察（基于失败事件）

### 洞察 1：可靠性=冗余度
当前系统对单一 LLM 提供商依赖过重。每日反思任务作为关键机制，必须确保执行可靠性。需要：
- 至少两个不同提供商的模型配置
- 自动故障转移逻辑
- 执行前健康检查

### 洞察 2：成本意识是自我保护
信用额度耗尽是系统健康的红灯。应该：
- 在 heartbeat-state.json 中追踪各服务配额使用情况
- 设置预警阈值（如余额<20%时提醒）
- 建立成本监控仪表板

### 洞察 3：关键任务需要降级策略
反思任务作为每日必做，应该设计：
- **全功能路径**：LLM 可用时正常执行
- **降级路径**：LLM 不可用时记录失败、发送告警、等待恢复后补执行
- **离线路径**：极端情况下的最小执行能力（如仅记录"无法反思"状态）
