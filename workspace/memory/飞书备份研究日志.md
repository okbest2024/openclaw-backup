## 1️⃣ 分析 Bitable 失败原因

### API 调用参数检查

**已使用的参数**：
```javascript
// 创建应用成功
feishu_bitable_create_app({ name: "备份方案测试应用" })
// → app_token: HWyEbCjxBafQJSsFejuctV6Sn1f

// 创建表格成功
feishu_bitable_create_field({
  app_token: "HWyEbCjxBafQJSsFejuctV6Sn1f",
  table_id: "tblKpNvNG2j47Npb",
  field_name: "BackupTime",
  field_type: 5,  // DateTime
  property: { format: "YYYY-MM-DD HH:mm:ss", isSort: true }
})
// → ❌ 400 错误
```

**错误信息**：
```
API 返回 400 Bad Request
field_type=5 (DateTime) 存在已知 bug
建议改用 Text 类型存储 ISO 8601 时间戳
```

### 结论
- ✅ Bitable CRUD 基础功能完全可用（create_app, create_field, create_record, list_records）
- ❌ DateTime 字段类型（field_type=5）不稳定，需规避
- ✅ 可使用 Text 替代 DateTime 或使用其他存储方案

### 技术细节发现

**API 行为**：
1. `create_table` 时 `row_size` 是初始行数（通常设 1 存放表头）
2. `write_table_cells` 可以写入任意范围的行，超出当前 row_size 会自动扩展
3. 表格结构在 `list_blocks` 返回的 table block 的 `children` 数组中，每个 cell block 对应一个单元格
4. cell block 的文本内容在 `block_type=2` 的 text run 中

**建议的读取表格实现**：
```javascript
// 伪代码
async function readTableData(docToken, tableBlockId) {
  const blocks = await feishu_doc({ action: "list_blocks", doc_token: docToken });
  const tableBlock = blocks.find(b => b.block_id === tableBlockId);
  const rows = [];
  const cells = tableBlock.children; // cell block IDs
  for (let i = 0; i < cells.length; i += columns) {
    const row = [];
    for (let j = 0; j < columns; j++) {
      const cellBlock = blocks.find(b => b.block_id === cells[i + j]);
      if (cellBlock && cellBlock.table_cell) {
        const textBlock = cellBlock.children.find(c => c.block_type === 2);
        row.push(extractText(textBlock));
      } else {
        row.push('');
      }
    }
    rows.push(row);
  }
  return rows;
}
```

**去重策略优化**：
- 使用 `filePath` 作为唯一键（而非文件名，避免冲突）
- 状态文件的 `backups` 映射存储 `{ relPath: { hash, backupTime } }`
- 首次全量备份，后续增量，降低 API 调用频率
