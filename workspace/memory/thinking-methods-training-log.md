### 隐含假设分解

**问题陈述中的隐含假设：**
1. 当前backup系统已用WAL实现事务化，但"稳定性"仍有缺口（日志中提及需要：哈希比对、智能追加、监控告警、恢复验证）
2. "稳定性"目标是"零丢失"，但未定义何为"丢失"（数据丢失？状态不一致？静默错误？）
3. "自验证"意味着系统能自动发现备份异常
4. "自恢复"意味着系统能自动修复备份问题
5. WAL事务化已解决写入可靠性，但完整性校验（hash comparison）仍是手动或缺失
6. "监控告警"需要独立的monitor组件，可能增加复杂度
7. "恢复验证"需要模拟恢复场景，可能影响生产系统
8. 当前backup策略是"全表覆盖写入"，已实现但可优化为"智能追加"（避免重复读取）
9. 这些改进需要新代码，而非配置调优
10. 升级过程需要保持现有backup服务不中断（向前兼容）
11. 完整性机制本身不能成为单点故障源
12. 最终系统应"自主运行"，无需人工周期性检查

**深层假设（未明说）：**
13. 备份完整性校验的"事实真相"是文件哈希（但文件内容可能合理变化，哈希不等=错误？）
14. 心跳状态的"正确状态"有单一版本，可diff对比
15. 历史记录的"正确性"可通过WAL日志完整性验证（但WAL自身也可能损坏）
16. 恢复验证（sandbox restore）是必要的，但频率未知
17. "智能追加"算法需要历史数据分析（但可能引入过度复杂化）
18. 监控告警阈值应自适应，而非固定规则
19. 系统需要"自主演化"能力，即策略随历史表现动态调整

### 基本原理识别

**A. 数据完整性基本定律：**
- **哈希不可逆性**：SHA256等密码学哈希，输入变化→输出变化（雪崩效应），已知输出无法反推输入
- **幂等性原理**：重复应用相同操作结果不变（WAL事务依赖此特性）
- **原子性保证**：文件rename在POSIX系统是原子的，但跨平台有差异（Windows ≠ Linux）
- **fsync可靠性**：调用fsync后数据落盘，但硬件缓存可能仍有风险（需要O_SYNC或硬件电池）

**B. 系统可靠性工程原理：**
- **Checksum验证**：传输/存储后重新计算校验和，与原始值对比
- **Write-Ahead Logging (WAL)**：先写日志再修改数据，确保可恢复性
- **故障隔离**：错误不应扩散，单点失败不影响整体
- **幂等性设计**：重复操作无害，便于重试
- **心跳检测**：定期检查系统健康状态
- **自愈系统**：检测→诊断→修复的闭环

**C. 备份系统特定原理：**
- **增量 vs 全量**：完整备份可校验，增量备份依赖链完整性
- **版本对比**：任意两个版本可diff，确定变化
- **状态快照**：某一时刻的内存/文件状态，可持久化
- **触发时机**：事件驱动 vs 时间驱动 vs 状态变化驱动
- **存储代价**：哈希值很小（32-64字节），但历史追踪需要空间
- **计算代价**：大文件哈希计算昂贵，但增量计算可优化

**D. 自验证系统的理论边界：**
- **不可验证的假设**：系统无法100%验证"程序逻辑正确性"（停机问题）
- **有限自验证**：可验证"数据未被篡改"（通过hash chain），但无法验证"数据语义正确"
- **自恢复的局限性**：只能恢复"已知状态"，无法恢复"未知的正确状态"
- **自指悖论**：验证系统本身的完整性需要外部信任根（bootstrapping问题）

**E. 监控与告警原理：**
- **阈值动态性**：静态阈值要么太敏感（误报），要么太迟钝（漏报）
- **历史基线**：健康系统的指标分布可建模，异常偏离可检测
- **告警疲劳**：过多无效告警导致忽略，需要精准度>95%
- **闭环反馈**：告警应触发修复，修复应更新学习模型

### 从原理重新推导

**步骤1：定义"零丢失"的精确含义**
从数据完整性定律出发，"丢失"必须可观测：
- 物理丢失：数据块损坏/丢失（hash mismatch）
- 逻辑丢失：数据覆盖错误（新数据被旧数据覆盖）
- 静默丢失：数据未备份但系统误报成功（状态不一致）
- 持久性丢失：WAL未fsync成功但标记为committed

因此，零丢失 = **所有丢失类型均可检测 + 检测到即自动修复 + 修复成功率100%**

**推导1**：需要**三重验证**：
1. WAL write + fsync → 确保事务持久性
2. Backup success + hash record → 确保备份数据完整性
3. State consistency check → 确保heartbeat-state与文件系统状态一致

**步骤2：从故障隔离原理推导架构**
单一组件不能承担全部验证职责（单点风险）。因此：
- **验证三独立**：WAL验证、文件验证、状态验证由三个独立进程/线程执行
- **数据源分离**：WAL文件、备份数据、heartbeat-state三方互不依赖
- **故障传播阻断**：任一方失败不应导致系统崩溃，应隔离并告警

**推导2**：需要**异构图谱**（heterogeneous witness）：
- Witness A: WAL完整性（log replay + checksum）
- Witness B: 文件系统哈希（periodic scan + diff）
- Witness C: 状态连续性（heartbeat检查 + session连续性）
任一witness失败 → 触发修复，无需其他witness确认

**步骤3：从自愈系统原理推导触发器**
自愈≠持续运行，而是**变化驱动**。
- 什么变化需要响应？WAL增长异常、备份失败、hash不匹配、heartbeat缺失
- 不应轮询（性能差），应由事件触发（WAL commit、backup finish、heartbeat）
- 但事件可能丢失（进程crash），因此需要**状态机**追踪"上次验证状态"

**推导3**：设计**状态变化监听器**（state-change detector）：
- 监听5个关键状态：`wal_seq`, `last_backup_hash`, `heartbeat_state`, `file_mod_time`, `system_uptime`
- 任意状态变化 → 触发对应验证链
- 验证通过 → 更新验证记录，不告警
- 验证失败 → 计算修复策略，自动执行或升级告警

**步骤4：从恢复验证原理推导风险**
"验证"通常意味着"尝试恢复并确认成功"，但这可能：
- 产生大量I/O（频繁sandbox restore）
- 影响heartbeat性能
- 破坏WAL幂等性（如果恢复过程修改原始数据）

**推导4**：使用**影子恢复**（shadow restore）：
- 将backup数据复制到临时目录
- 在隔离进程内执行"假恢复"（不修改原始WAL/state）
- 验证JSON语法、字段完整性、哈希一致性
- 仅当验证失败且无法自动修复时，才触发"真实恢复"（atomic switch）

**步骤5：从"智能追加"优化推导**
当前"全表覆盖"的代价：
- 每次backup读取整个表格（mem+IO）
- 写入全表（网络传输）
- 如果表格行数多（>1000），延迟可能>5s

优化方向：
- 增量读取：只读取上次备份后的新行（需要记录lastRowIndex或lastTimestamp）
- 增量写入：append新行而非覆盖全表（但云文档表格API无append，只能覆盖）
-cloud doc table的局限：**必须全表覆盖写入**，无行级别append

**推导5**：智能追加**不可行**，必须在客户端实现"读取-合并-覆盖"：
1. Read all rows (批量获取，≤500行/页，可能需要分页)
2. Filter out existing files by filename
3. Append new rows
4. Write full table back
这不是"智能"，是**必要流程**。优化方向：
- 缓存本地已知哈希，避免重复读取表格
- 并行分页读取（如果行数>500）
- 使用Markdown表格替代可交互表格（无格式，纯文本append可能支持？但API未知）

**步骤6：从监控告警原理推导告警疲劳解决方案**
如果每次backup失败都告警 → 告警风暴（如果backup每4小时失败）
需要：
- 失败抑制：同一错误连续发生，只告警一次
- 严重性分级：hash mismatch（critical） vs 网络超时（warning）
- 自愈优先：告警前尝试自动修复3次
- 上下文丰富：告警携带"最后成功时间、错误计数、修复建议"

**推导6**：设计**三级响应机制**：
1. Auto-fix（0级）：自动尝试3次，指数退避（立即、+2min、+5min）
2. Self-alert（1级）：修复失败，系统内记录并通知主会话（但非紧急）
3. Human-alert（2级）：连续失败>3次或影响核心数据（heartbeat-state/MEMORY.md），立即通知主人

**步骤7：从自指悖论推导信任根**
备份系统验证自身，需要外部信任锚点：
- 初始部署时的heartbeat-state的hash是可信的（主人在场）
- 此后所有状态变更应通过WAL追踪
- 如果WAL损坏，需要**恢复点（Recovery Point）**——即上次成功备份的云端表格作为真相源

**推导7**：建立**三源一致性协议**：
- Source A: WAL (transaction log, authoritative)
- Source B: Cloud table (backup records, queryable)
- Source C: Local file (current state, ephemeral)
验证规则：B与C的hash必须匹配A中最新committed transaction。如不匹配，按优先级选择：

```
优先级：A > B > C  (WAL > Cloud > Local)
- 若A损坏但B完好：从B重建A（replay records）
- 若B损坏但A完好：从A重建B（re-run backup）
- 若A、B均损坏：从C重建（local file可能丢失，但尝试恢复）
- 若C损坏但A、B完好：从B恢复C（restore from cloud）
```

### 新方案

**方案名称：零丢失状态见证协议 v3.0（三源一致性+自适应监控）**

**核心理念**：备份系统的"存在"不是为了"备份"，而是为了"见证状态"——通过三重存在（WAL、Cloud、Local）的持续共振，让状态自证完整，让异常自露破绽。

**架构三层**：
1. ** witnessing-core**: 状态变更捕获、WAL事务、hash计算、事件发布
2. ** adaptive-monitor**: 状态变化监听、策略选择、执行验证、自修复
3. ** resilience-layer**: 告警分级、恢复点管理、跨源一致性协议

**核心组件**：

#### A. Triple Witness Engine ( witnessing-core )
- **Event Bus**: 发布`backup:success`, `wal:commit`, `heartbeat:change`, `file:modify`
- **Hash Ledger**: 记录每个备份文件的hash_last_success, hash_current
- **WAL Committer**: 所有状态变更进入WAL，fsync后标记committed
- **Checkpoint Snapshot**: 每10次backup生成完整state snapshot存入Cloud

#### B. Adaptive Monitor ( adaptive-monitor )
- **State Change Detector**: 订阅Event Bus，检测关键状态变化（WAL长度突增、hash不匹配、heartbeat超时）
- **Strategy Engine**: 根据变化类型选择验证策略（快速校验:hash vs 完整性校验:sandbox restore）
- **Recovery Planner**: 失败时评估修复选项，选择最安全且最低成本的路径
- **Metrics Collector**: 记录验证成功率、修复耗时、告警准确率，用于训练策略

#### C. Resilience Layer ( resilience-layer )
- **Three-Source Consistency Protocol**: 实现A>B>C优先级恢复规则
- **Recovery Point Manager**: 保存最近3个有效checkpoint，支持时间点恢复
- **Alert Router**: 三级路由（auto-fix → self-alert → human-alert），带抑制和降级
- **Chaos Injector**: 每周自动注入故障，测试自愈能力（生产系统可关闭）

**关键机制**：

1. **智能追加优化**（尽管API限制，仍可减少读取）：
   - 本地缓存backup records（JSON数组，包含filename+hash+timestamp）
   - backup前先对比缓存，只上传新增rows
   - 但write_table_cells仍需全表覆盖，客户端做diff合并
   - **优化点**：缓存有效期24h，过期后强制全表读取重建

2. **自验证流程**：
   ```
   on backup:success → 
     1. compute hash of uploaded data (verify cloud received correctly)
     2. compare with hash_ledger entry (detect mismatch)
     3. if mismatch: 
           - attempt 1: re-read cloud table, verify again
           - attempt 2: re-compute local file hash, compare with cloud
           - attempt 3: check WAL transaction integrity
         else:
           - update hash_old = hash_new, mark as verified
   ```

3. **恢复验证（非侵入）**：
   ```
   every 24h →
     1. pick latest checkpoint
     2. clone cloud table data to temp file
     3. validate JSON schema of heartbeat-state/MEMORY.md
     4. compute hash and compare with ledger
     5. if success: log "recovery validation ok"
     6. if fail: trigger recovery protocol immediately
   ```

4. **自适应阈值**：
   - 初始阈值：backup成功<99.9%触发告警
   - 学习模式：收集30天数据后，计算成功率分布p(μ=99.95%, σ=0.05%)
   - 动态调整：如连续7天>99.98%，阈值提升至99.95%；如波动剧烈（σ>0.1%），阈值放宽至99.8%

**预期效果**：
- 短期（1天）：WAL+hash验证上线，首次异常检测成功
- 中期（7天）：自适应阈值生效，误报率<1%，自动修复成功率>99%
- 长期（30天）：系统达到"遗忘"状态——主人不再感知备份存在，但数据完整性达99.999%

**风险与缓解**：
- **WAL增长失控**：设置WAL max size=100MB，archival自动清理（success事务保留30天，failed保留90天）
- **fsync性能**：wal-manager.js已异步，但critical path需sync → 使用`fsync(fd, 3)`（linux）或`FlushFileBuffers`（Windows）
- **云API限流**：write_table_cells频繁可能触发rate limit → 指数退避+批量操作
- **三源分歧**：A/B/C同时不一致时，选择恢复点而非自动修复，避免恶性循环
- **自适应误判**：初期阈值保守（固定99.9%），积累30天数据后启用自适应

### 新方案 vs 原方案的对比

| 维度 | 原方案（WAL事务化 v2.0） | 新方案（三源一致性 v3.0） |
|------|------------------------|------------------------|
| **完整性验证** | 仅WAL内checksum，无数据hash | hash ledger + 云数据对比 + 本地验证 |
| **错误检测** | WAL损坏时告警，但静默覆盖错误不检测 | 检测hash mismatch → 发现静默错误 |
| **自恢复能力** | 从WAL重放，但WAL可能全毁 | 三源优先级恢复，支持云源重建 |
| **监控粒度** | 成功/失败二元状态 | 多维指标（成功率、延迟、校验一致率、修复率） |
| **告警策略** | 失败即告警 → 告警疲劳 | 三级路由 + 自动修复优先 + 自适应阈值 |
| **进化能力** | 固定逻辑 | 策略学习引擎（基于历史表现调整阈值和策略） |
| **复杂性** | 中等（3个核心组件） | 高（7个组件，但关注点分离，可独立测试） |
| **向前兼容** | 需修改backup脚本 | 可并行运行，逐步迁移 |

**优势**：
1. **零丢失可验证**：通过hash ledger，任何数据变化都被记录和验证
2. **自恢复不依赖单一源**：打破"所有鸡蛋在WAL篮子"的风险
3. **告警精准性**：99%以上失败可自动修复，仅临界失败通知主人
4. **适应性**：系统随运行时间增长，阈值和策略更精准
5. **安全边界清晰**：三源优先级（A>B>C）提供确定性恢复决策树

**代价**：
1. **开发量**：约2-3天实现完整方案（vs 现状已投入的WAL基础）
2. **运行时开销**：每次backup多2次hash计算（增量<50ms）、额外Event Bus发布
3. **存储开销**：hash ledger约100bytes/backup × 6次/天 × 30天 ≈ 18KB/月，可忽略
4. **WAL管理**：需要background job归档旧WAL，防止磁盘占满

**关键差异点**：
- **范式转变**：从"尽力而为的备份"到"银行级状态见证"
- **验证内嵌**：备份不是"完成即结束"，而是"完成+验证+记录"的原子链条
- **系统韧性**：即使WAL和cloud双重损坏，仍可能有local file可恢复（三源设计）

**时间：** 2026-03-29 05:35 (Session 710 cron)
**训练问题：** 云文档表格备份的性能优化 —— 如何减少全表覆盖写入带来的网络传输和延迟开销？

### 隐含假设分解

**问题陈述中的隐含假设：**
1. 当前backup系统使用feishu_doc的create_table_with_values/write_table_cells，采用全表覆盖模式
2. "性能问题"主要表现为网络传输量大（表格越大，每次传输整个表格）和延迟高（写入时间随行数线性增长）
3. 云文档API只支持全表覆盖写入，不支持行级别append操作（这是已确认的限制）
4. 读取表格数据可行（list_blocks解析表格），读取成本也需要考虑
5. 性能瓶颈可能在"写入"而非"读取"（因为写入需要传输完整数据）
6. 目标不是"无限扩展"，而是"在可接受延迟内（如<2秒）支持至少1000行备份索引"
7. 当前6个目标文件，每4小时备份一次，未来可能扩展到更多文件
8. 缓存机制可能引入复杂度（缓存失效、一致性、存储空间）
9. 优化应在现有API约束下进行，不能要求飞书改变API
10. 优化不应降低数据的可靠性或完整性（hash验证必须保留）
11. 优化算法本身不应成为新的单点故障或性能瓶颈
12. 客户端内存占用是可接受的（<100MB），但不应无限制增长

**深层假设（未明说）：**
13. "性能"主要指端到端耗时（从决定backup到写入完成），而非单一阶段的耗时
14. 网络带宽是相对充裕的，瓶颈可能在飞书服务器处理时间而非传输时间
15. 表格行的定义是"每次backup新增的一行记录"，行数=备份次数×文件数量
16. 表格的"列"是固定的（8列），列数不增长，因此列宽固定
17. 读取表格时，所有行都需要被读取（全表扫描），无论是否有过滤条件
18. 飞书API的分页查询是可靠的（不会遗漏或重复），但可能存在延迟
19. 客户端可以安全地并行请求（分页读取），不会触发API限流
20. "智能追加"在API限制下必须降级为"读取-合并-覆盖"，但读取可以增量（只读新增）
21. 本地缓存需要持久化（内存缓存重启丢失），可能用文件或独立存储
22. 性能优化优先级：用户感知延迟 > 网络流量 > 服务器负载 > 客户端内存

### 基本原理识别

**A. 算法复杂度理论：**
- **时间复杂性**：全表操作 O(n)，分页读取 O(n)（但可并行常数因子加速）
- **空间复杂性**：全表存储在内存 O(n×m)（n行×m列），增量策略可降低为 O(Δn×m)
- **并行加速**：k个并行请求可将时间降低为 O(n/k + overhead)，但受限于API速率限制
- **缓存原理**：缓存命中 O(1)，缓存未命中 O(n)，缓存一致性需要额外开销

**B. 网络传输原理：**
- **带宽-延迟积**：网络带宽有限，数据量大时传输时间主导
- **TCP慢启动**：新连接传输效率低，重复连接比长连接开销大
- **分块传输**：大数据分块传输可避免超时，但增加请求次数
- **压缩权衡**：文本数据压缩率有限（JSON可能30-50%），压缩/解压消耗CPU

**C. 飞书API特性（基于已验证行为）：**
- **write_table_cells**：全表覆盖写入，更新整个表格内容
- **list_blocks**：获取文档所有block，表格cell数据包含在table块中（可能分页？）
- **无append端点**：无法在表格末尾添加单行，必须重写整表
- **无行级别更新**：无法更新特定行而保留其他行（除非先读取再重建）
- **API限流策略**：未知，但需要假设存在（如100 req/min）
- **原子性**：write_table_cells是原子操作（整表替换），要么成功要么失败）

**D. 数据结构与内存管理：**
- **数组操作**：JavaScript数组concat/filter时间复杂度 O(n)
- **内存拷贝**：大数组复制消耗内存和时间
- **垃圾回收**：频繁创建大数组可能触发GC pause
- **增量更新**：只追踪变化部分，内存 O(changes) 而非 O(total)

**E. 一致性模型：**
- **最终一致性**：缓存与云表格可能存在短暂不一致
- **强一致性需求**：backup索引记录必须准确反映已备份的文件，不能丢失或重复
- **冲突解决**：如果两个程序同时backup，后写入覆盖先写入（需避免并发）
- **事务边界**：读-合并-写必须是一个逻辑单元，中间失败需要重试

### 从原理重新推导

**步骤1：确定性能瓶颈的真实位置**
从网络传输原理，端到端延迟 = 读取延迟 + 处理延迟 + 写入延迟 + 网络延迟
- 读取延迟：list_blocks获取表格数据，数据量 = 行数×列数×平均字符数
- 写入延迟：write_table_cells传输整个表格，数据量≈读取数据量 + 新行数据
- 假设当前表格有300行（4小时×6文件×约1.3天），每行约200字节 → 60KB数据
- 60KB在现代网络（1Mbps+）传输时间 < 1秒，因此**瓶颈可能在飞书服务器处理时间**（数据库写入、索引更新）

但表格增长趋势：
- 每天新增约36行（4小时×6×6次/天）
- 30天后约1080行，216KB
- 一年后约13140行，2.6MB

**推导1**：问题本质不是"当前慢"，而是"增长无界" → 需要**限制表格历史长度**，而非仅优化单次传输。

**步骤2：从算法复杂度推导可行优化**
全表覆盖写入必须 O(n)，但n可以缩小：
- 方案A：只保留最近N行（滚动窗口），丢弃旧行
- 方案B：归档旧行到另一表格或文档，主表格只保留近期
- 方案C：分表存储（按时间分表），减少单表行数

但需求要求"完整备份历史"（用于恢复、审计、去重），因此不能简单丢弃。

**推导2**：**不变式**：必须保留所有历史行 → 无法避免 O(n) 读取和写入 → 必须接受随着n增长，延迟线性增长。

**步骤3：从缓存原理推导绕过读取的可能性**
如果我们可以不读取现有表格，就能写出新行，那就不需要 O(n) 读取：
- 需要知道：表格当前最后一行索引或行数
- 但write_table_cells需要完整2D矩阵，你必须提供所有行（包括已存在的行）
- 因此，**你至少需要重建已存在的行数据结构，即使你不知道其内容**

除非飞书API支持"append mode"或"partial update"，否则**读取是不可避免的**。

**推导3**：优化核心不是"避免读取"，而是**减少读取的频率和数据量**：
- 频率：从"每次backup都读取" → "仅当缓存失效时才读取"
- 数据量：从"读取全表" → "读取增量（新行）+ 本地缓存（旧行）"

**步骤4：从并行原理推导读取加速**
list_blocks是否支持分页？文档API通常返回所有blocks一次（如果文件不大）。
- 如果表格行数<1000，单次请求可行
- 如果行数>1000，可能需要分页读取（如果API支持）
- 并行读取无意义（单次请求已获取全部）

**推导4**：读取优化只能依赖**缓存**，无法依赖并行。

**步骤5：从一致性原理推导缓存策略**
缓存必须保证：当写入成功时，缓存立即更新为最新完整表格状态。
- 读取 → 合并新行 → 写入 → 成功 → 更新缓存
- 如果写入失败，缓存不能更新（保持旧状态），下次重试仍基于旧状态+新数据（幂等）
- 如果缓存文件损坏，需要回退到全表读取（fallback）

**推导5**：**缓存作为权威副本**（authoritative copy）的本地镜像：
- 缓存格式：JSON数组，每行对象 {filename, type, path, time, size, hash, status, note}
- 缓存元数据：last_update_time, rowCount, cloud_table_version（如果表格有version）
- 缓存写入时机：cloud write成功后的原子更新（先写temp，再rename）
- 缓存读取时机：backup开始时，检查缓存有效性（时间戳、文件存在）

**步骤6：从时间复杂性推导"缓存命中"收益**
假设：
- 全表读取延迟 R_full = 读取时间 + 解析时间 ≈ 500ms（1000行）
- 全表写入延迟 W_full = 序列化 + 传输 + 写入 ≈ 1000ms
- 缓存命中延迟 R_cache = 读取缓存文件 + 解析 ≈ 50ms
- 新行数据量 ≈ 6行/次 × 200字节 ≈ 1.2KB，计算时间≈5ms
- 合并（concat）延迟 M = O(n) 数组拼接，1000行≈10ms

场景对比：
| 场景 | 读取 | 合并 | 写入 | 总延迟 |
|------|------|------|------|--------|
| 无缓存（每次全读） | 500ms | 0 | 1000ms | 1500ms |
| 有缓存（25%缓存失效率） | 50ms×0.75 + 500ms×0.25 = 175ms | 10ms | 1000ms | 1185ms |

**推导6**：缓存命中率需要 > 80% 才能将总延迟降低到 1秒以内（当前1500ms→目标<1000ms）。
- 如果backup每4小时一次，缓存失效可能是文件损坏或人为清理
- 设计目标：缓存失效概率 < 5%（即20次backup仅1次失效）

**步骤7：从网络传输原理推导数据量缩减**
当前传输的表格内容包括：
- 表头行（固定8列标题）
- 所有历史行（每行8列数据，text格式）
- 传输数据量 ≈ (1 + n) × 8 × avg_col_width

压缩思路：
- 列数固定，可编码列名只传输一次（但write_table_cells需要完整2D矩阵，无法修改协议）
- 数值型数据（大小、时间戳）可压缩，但文本压缩率有限
- 如果使用二进制协议可能更优，但API限制只能传JSON

**推导7**：**数据量优化不可行**，因为API要求完整的JSON矩阵，必须传输所有字符。
唯一数据量减少途径：减少n（行数）→ 回到步骤1的**历史滚动窗口**策略。

**步骤8：从一致性原理推导"历史滚动窗口"的可行性**
如果我们将表格分为：
- **主表格**（主表）：保留最近M行（例如最近30天，约1080行）
- **归档表格**（副表）：保留全部历史行，只读，定期归档（如每月一次）

backup流程：
1. 读取主表格（行数受限，<1100行）
2. 合并新行 → 写入主表格
3. 主表格达到M行阈值时，触发归档：
   - 复制当前主表格全部内容到归档表格
   - 清空主表（或保留最近K行作为种子）
   - 重置缓存和hash ledger

对恢复的影响：
- 恢复时可能需要查询归档表格获取更早的历史记录
- 但heartbeat-state、MEMORY.md等核心文件恢复只需最近一次备份，不需要全历史

**推导8**：**滚动窗口可接受**，因为：
- 备份索引的主要用途是"找到最新备份"，而非"查看所有历史"
- 完整历史可通过归档表格保留（访问频率极低，可手动查询）
- 主表格大小可控，性能稳定

### 新方案

**方案名称：云文档表格性能优化 v1.0 —— 本地缓存 + 滚动窗口双引擎**

**核心原则**：性能优化不是无限压缩传输量，而是在"数据完整性"与"操作延迟"之间找到实用平衡。通过**缓存命中**减少读取，通过**滚动窗口**限制写入规模。

**架构设计**：


**关键组件详解**：

#### 1. CacheManager（缓存管理器）
- **缓存文件路径**: `memory/feishu-backup-cache.json`
- **数据结构**：
  ```json
  {
    "meta": {
      "rowCount": 342,
      "tableBlockId": "doxcnwhyXhKB6ORGWeAHoW6vlJf",
      "docToken": "GaDhdogBhoQWRQx5lG4cpyQknUb",
      "lastSync": "2026-03-29T05:30:00Z",
      "cacheVersion": 1
    },
    "rows": [
      ["heartbeat-state.json", "配置文件", ...],
      ...
    ]
  }
  ```
- **有效性检查**（每次backup开始时）：
  1. 文件存在且可读
  2. JSON格式有效
  3. `meta.rowCount` >= 0 && `meta.rowCount` <= 当前表格实际行数（如果表格行数增加，缓存只可能失效，不会导致数据丢失）
  4. `meta.tableBlockId` 与当前配置一致（防止缓存污染）
  5. `lastSync` 在24小时内（防止陈旧缓存）

- **缓存策略**：Write-through（写入cloud成功后立即写入缓存文件）
- **缓存更新**：原子操作（先写`.tmp`文件，再`rename`到正式文件）

#### 2. WindowManager（滚动窗口管理器）
- **配置**：
  ```javascript
  const MAX_ROWS = 1200;  // 主表最大行数（约30天历史）
  const ARCHIVE_TRIGGER_RATIO = 0.9; // 达到90%触发归档
  const ARCHIVE_TABLE_NAME = "Backup Archive ({year}-{month})";
  ```
- **检查时机**：合并新行后，写入cloud前
- **触发条件**：`newRowCount >= MAX_ROWS * ARCHIVE_TRIGGER_RATIO`
- **归档动作**：
  1. 创建归档表格（如果本月归档表不存在）
  2. 将当前主表所有行（包括新行）追加到归档表（注意：归档表同样需要全表覆盖写入，但频率极低）
  3. 清空主表（保留表头）
  4. 重置缓存（清空rows数组，保留meta结构）
  5. 记录日志："Archive triggered: {newRowCount} rows archived to {archiveTableId}"

**注意**：归档操作可能耗时（写入1200行到归档表），应在低峰期（如凌晨）执行，或延迟到下次backup周期。

**简化方案**（第一版）：不自动归档，只是**停止向主表添加新行**，并在日志中告警："主表已达{MAX_ROWS}行上限，请手动执行归档或增加MAX_ROWS"。保持简单。

#### 3. 读取流程优化
```javascript
async function getExistingRows(cacheManager) {
  // 尝试快速路径：缓存有效
  if (await cacheManager.isValid()) {
    return await cacheManager.getRows(); // 50ms
  }
  
  // 慢速路径：从cloud读取
  console.log("Cache miss/invalid, reading from cloud...");
  const rows = await readTableFromCloud(docToken, tableBlockId, columnSize);
  await cacheManager.setRows(rows); // 更新缓存
  return rows;
}
```

#### 4. 写入流程优化
```javascript
async function writeWithRollingWindow(rows, cacheManager) {
  const currentCount = rows.length;
  
  // 检查滚动窗口
  if (currentCount >= MAX_ROWS * ARCHIVE_TRIGGER_RATIO) {
    await handleArchive(rows); // 可能抛出错误，但不阻断backup（告警即可）
    // 归档后清空主表数据，只保留表头
    rows = [rows[0]]; // 假设第0行是表头
  }
  
  // 写入cloud（全表覆盖）
  await feishu_doc({
    action: "write_table_cells",
    doc_token: docToken,
    table_block_id: tableBlockId,
    values: rows
  });
  
  // 更新缓存
  await cacheManager.setRows(rows);
}
```

#### 5. 缓存失效的fallback
- 如果缓存文件损坏：删除并重建（从cloud读取）
- 如果cloud读取失败：尝试使用缓存（即使过期）作为fallback，但记录警告
- 如果两者都失败：backup失败，告警

### 预期效果

**当前状态**（无缓存，300行）：
- 读取延迟：500ms
- 写入延迟：1000ms
- 总延迟：~1500ms

**优化后**（缓存命中率95%，1200行上限但95%命中缓存）：
- 读取延迟：50ms (95%) / 500ms (5%) = 75ms 平均
- 写入延迟：1000ms（不变，但1200行后触发归档，写入最大1200行）
- 总延迟：~1100ms 平均

**未来扩展**（到1000行）：
- 读取延迟（95%命中）：50ms
- 写入延迟（1000行）：~1200ms（传输量翻倍）
- 总延迟：~1300ms

**归档后的重置效果**：归档后主表清空，回到约100行，延迟重回~600ms

**性能提升总结**：
- 延迟降低：~25%（从1500ms→1100ms）
- 主要收益：减少读取时间（500ms→75ms）
- 写入时间无法优化（API限制）
- **长期稳定性**：主表行数不会无限增长，性能不会无限恶化

### 与原方案的对比

| 维度 | 原方案（每次全读全写） | 新方案（缓存+滚动窗口） |
|------|----------------------|----------------------|
| **读取延迟** | 每次500ms（线性增长） | 缓存命中50ms，仅cache miss时500ms |
| **写入延迟** | 每次1000ms（线性增长） | 相同，但行数受MAX_ROWS限制 |
| **内存占用** | 读取时存储全表 O(n) | 缓存常驻内存 O(n)，但n有上限 |
| **网络流量** | 每次下载全表 + 上传全表 | 缓存命中时只上传不下载；miss时下载+上传 |
| **可扩展性** | n增长→延迟无限增长 | n增长到阈值后触发归档，重置n |
| **实现复杂度** | 简单（1个函数） | 中等（缓存管理+窗口检查+归档逻辑） |
| **故障风险** | 低（无状态） | 缓存文件损坏风险（需fallback） |
| **数据完整性** | 保证（每次校验hash） | 相同，但缓存需额外验证 |

**优势**：
1. **性能可预测**：行数上限1200，延迟峰值可预期（~1500ms），不会随时间无限恶化
2. **缓存加速显著**：95%命中率下，读取延迟降低90%
3. **归档机制保证长期运行**：一年的历史数据不会堆积在主表，可分卷存储
4. **向后兼容**：缓存文件新增，但旧版本仍可工作（只是性能差）

**代价**：
