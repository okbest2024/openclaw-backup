### 基本原理识别
**物理/数学/已验证事实基础：**

- **原理A（约束优化）：** 给定有限时间资源（48小时），最大化可靠性提升 = 选择边际收益最高的任务子集
  - 数学基础：0-1背包问题变体，目标函数：∫(可靠性提升, 0→48h) d(t) 最大化
  - 约束：∑(任务时间) ≤ 48h，任务间依赖（DAG）

- **原理B（递减收益）：** 任务集合的可靠性收益存在递减规律，前20%任务可能贡献80%收益（帕累托）
  - 事实基础：多个系统改进场景中，关键路径的少数任务确实决定大部分结果
  - 推论：应识别"收益-时间比"最高的任务

- **原理C（风险累积）：** 单点故障风险随时间线性累积（每日概率p），而系统改造有一次性风险
  - 事实：OpenRouter已degraded，完全失效风险每日递增（配额=0）
  - 数学：累积失效概率 = 1 - (1-p)^n，n为天数

- **原理D（幂等性定义）：** 任何新建系统（Fallback）的可靠性 < 成熟系统（当前心跳/备份）直到经过验证期T
  - 事实：新代码未经真实故障测试，初始可靠性通常只有60-80%
  - 推论：直接替换关键路径风险极高，应并行运行 + 渐进流量切换

- **原理E（任务网络结构）：** 任务的真实依赖关系不是链式而是有向无环图（DAG），存在关键路径
  - 事实：executeWithFallback.js 可独立测试；cron集成可逐步进行；监控可后置
  - 工具：临界路径法（CPM）可识别必须序列部分

### 从原理重新推导
**从零推导最优2日方案：**

**Step 1: 形式化问题**
- 变量：S = 任务集合 {A,B,C,D,E} 对应5项任务
- 参数：t_i = 任务i耗时（小时），r_i = 任务i带来的可靠性提升（单位：失败概率降低%）
- 目标：max ∑(r_i)  subject to ∑(t_i) ≤ 48h 且依赖约束
- 约束：某些任务必须完成后才能启动后续（如：集成→测试）

**Step 2: 任务分解与耗时估计（先验）**
任务A: executeWithFallback集成 (3个cron) -> 估计每个2h，共6h
任务B: provider-quota-tracker.json 创建 -> 估计1h
任务C: executionCapabilities 设计 -> 估计2h（需要思考）
任务D: 端到端故障模拟测试 -> 估计4h（需要环境准备）
任务E: 全量部署 + 监控 -> 估计4h

总时间 = 6+1+2+4+4 = 17h，总收益 = r_A + r_B + r_C + r_D + r_E

**Step 3: 识别收益函数 r_i**
- r_A = **最高**（直接消除单点故障，关键路径）
- r_B = 中等（成本失控风险，但紧急度次于可用性）
- r_C = 中等低（能力边界模糊会导致fallback质量风险）
- r_D = **关键**（验证系统有效性，无验证则集成无意义）
- r_E = 高（但若测试未通过，部署反而放大风险）

**Step 4: 依赖分析（DAG）**
```
          C (能力设计)
         /
A (集成) → D (测试) → E (部署)
         \
          B (成本监控) - 无强依赖，可并行
```

- 关键路径：A → D → E （时间：6+4+4=14h）
- C 必须与 A 同时或前置（因集成需要能力标记）
- B 可独立并行

**Step 5: 收益-时间比（ROI）计算（归一化）**
假设 r_A=40%, r_B=20%, r_C=20%, r_D=30%, r_E=25%（总收益可>100%，因重叠）

ROI_A = 40/6 ≈ 6.67 %/h
ROI_B = 20/1 = 20 %/h  ⭐⭐⭐⭐⭐
ROI_C = 20/2 = 10 %/h
ROI_D = 30/4 = 7.5 %/h
ROI_E = 25/4 = 6.25 %/h

**Step 6: 帕累托分析**
- B 的 ROI 最高，且依赖度低 → 应优先并行执行
- A 的绝对收益最高，是关键路径起点 → 必须做
- C 必须在集成前完成，但耗时少 → 可并行或前置
- D 和 E 收益依赖前置完成度

**Step 7: 2日（16小时可用工作时间）时间盒约束**
- 假设每日有效工作时间 8h（实际可能更少），总有效 = 16h
- 如果严格按照原计划17h，已经超时1h
- 需要**时间压缩**或**范围削减**

**Step 8: 范围削减分析（最小可行Phase 1）**
定义 Phase 1 "完成"的最小充分条件：
- 必需（Must）：A1（1个cron集成）+ D（测试验证）—— 验证fallback机制本身有效
- 重要（Should）：A2,A3（剩余cron）—— 覆盖更多关键任务
- 次要（Could）：B（成本监控）、C（能力标记）、E（全量部署）—— 可延后或简化

**Step 9: 重新权衡方案（2日极限冲刺）**
方案1（原计划）：6+1+2+4+4=17h → 不可行（超1h）

**Step 10: 考虑风险不对称性**

### 新方案

**目标重新定义**：不是"所有任务100%完成"，而是"关键cron任务获得验证过的fallback保护，置信度≥90%"
