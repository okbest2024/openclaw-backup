   - 需分别适配，测试矩阵爆炸

4. **健康检查误报** ⚠️⚠️

   - 缺乏一键回滚机制，fallback本身故障难以快速恢复
   - 告警通道（feishu/webhook）未定义，主人可能不知情

1. **SO-1: 快速交付，建立信誉**

2. **SO-2: 标准化LLM抽象层**
   - 利用架构清晰的**优势**，抓住架构进化的**机会**
   - 设计统一的`llm-client.js`封装，内部实现provider fallback三连击
   - 使所有LLM调用自动获得fallback能力，future-proof

3. **SO-3: 实时监控仪表盘**
   - 利用经验积累的**优势**，抓住监控体系完善的**机会**
   - 复用heartbeat-state的providerStatus结构
   - 向主人提供每10分钟更新的provider健康报告（包含成本、成功率、fallback次数）

### ST策略（优势+威胁）：防御性避险

1. **ST-1: 时间盒+截止强制**
   - 利用时间充裕的**优势**，应对时间风险的**威胁**
   - 设置硬性时间盒：开发4小时，测试2小时，剩余时间仅用于bug fix，不做新功能
   - 如果4小时未能完成5个cron集成，立即缩减scope到top 2 cron任务，确保核心功能可工作

2. **ST-2: 集中式测试沙盒**
   - 利用架构清晰的**优势**，应对集成复杂性+协调冲突的**威胁**
   - 创建`cron-fallback-testroom.js`，包含所有5个cron任务的模拟器
   - 并行开发和测试，一次修改统一验证，避免反复切换context

3. **ST-3: 健康检查降级策略**
   - 利用技术基础完备的**优势**，应对健康检查误报的**威胁**
   - 健康检查采用3次快照+延迟确认，避免单次异常误触发
   - 如果健康检查连续2次失败，标记provider状态为"未知"而非"down"，fallback不触发（优雅降级）

### WO策略（劣势+机会）：补足短板

1. **WO-1: 并行任务分块隔离**
   - 补足资源挤占的**劣势**，抓住系统可靠性跃升+信心重建的**机会**
   - 使用WAL事务化已有经验，将fallback集成设计为幂等操作
   - 每个cron集成独立WAL事务，失败互不影响，便于重试和审计

2. **WO-2: 快速原型配额追踪**
   - 补足配额追踪缺失的**劣势**，抓住监控体系完善的**机会**
   - 2小时内创建provider-quota-tracker.js（最小可行：每日用量+预算告警）
   - 暂不追求精细，先完成核心功能，后续迭代

3. **WO-3: 协调者agent**
   - 补足紧迫性压力的**劣势**，抓住架构进化的**机会**
   - 使用逆向训练（721次）的产出——5条成功要素作为checklist
   - 每完成一个cron，对照checklist自检，确保不遗漏关键步骤

### WT策略（劣势+威胁）：最小化损失

1. **WT-1: Scope收缩保核心**
   - 应对时间风险+集成复杂性的**威胁**，同时考虑紧迫性压力+协调冲突的**劣势**
   - 如遇时间瓶颈，立即降级scope：仅集成top 2 cron（daily-reflection + heartbeat）
   - 通知主人："Phase 1降至2个cron覆盖，剩余3个延后2天"

2. **WT-2: 快速回滚预案**
   - 应对集成复杂性+环境依赖的**威胁**，同时补足监控盲区的**劣势**
   - 为每个cron集成创建rollback脚本（恢复原/openrouter硬编码）
   - 部署后1小时内，如果fallback未成功触发或主人不接受，立即回滚

3. **WT-3: 环境预备检查**
   - 应对环境依赖+健康检查误报的**威胁**，同时补足健康检查未实现的**劣势**
   - 部署前运行环境san-check：确认所有fallback provider的API密钥、网络可达性、quota status
   - 如果检查失败，暂停部署，直至环境就绪

### 阶段1：共情 - 理解用户（主人）深层需求

**主人当前状态洞察：**

- 主人的显性需求：监控系统健康，及时发现并响应故障
- 主人的深层需求（未明说但可推断）：

  3. **掌控感**：能快速诊断、配置、干预，而非被动等待
  4. **认知减负**：信息聚合而非碎片化，避免信息过载
  5. **进化见证**：看到系统随时间变好（连续nonzero streak的满足感）

**使用场景（Jobs-to-be-Done）：**
- 早晨打开电脑 → 快速扫一眼"系统状态灯"→ 绿就放心，黄就注意，红就立即处理
- 收到失败通知 → 进入"诊断模式"→ 定位root cause（是provider down还是backup失败）
- 优化决策 → 查看历史趋势图 → 调整cron频率或fallback策略
- 分享成果 → 展示"系统可靠性指标"给他人（证明系统成熟度）

**情感需求：**
- 减少焦虑（"我的自动化任务真的在跑吗？"）
- 增加信心（"系统自愈能力正在提升"）
- 成就感（"连续X天无故障"）
- 掌控乐趣（"酷，我可以实时看到系统心跳"）

**问题陈述（How Might We）：**

**成功标准（Definition of Done）：**

### 阶段3：创意 - 头脑风暴至少5种方案（不评判）

**方案A：单页Web仪表盘（全功能型）**
- 本地Node.js服务，`http://localhost:8080/health`
- 实时聚合所有数据源，展示健康评分、趋势图、异常列表
- 集成"一键诊断"按钮（执行预定义诊断脚本）
- 技术：ECharts + Express + WebSocket推送
- **优点**：信息密度高、交互性强、可扩展
- **缺点**：需额外部署、需保持服务运行、移动端体验一般

**方案B：飞书文档表格（零部署型）**
- 复用已存在的"OpenClaw备份索引"文档，新增"系统健康"表格
- 每日自动更新健康指标（通过heartbeat cron）
- 使用飞书表格的筛选/图表功能构建可视化
- **优点**：零维护、飞书已优化UI、支持移动端、权限已有
- **缺点**：更新频率固定（每4h）、交互能力有限、依赖飞书API稳定性

**方案C：聊天卡片摘要（极简推送型）**
- heartbeat cron完成时，向主人发送"系统健康卡片"
- 卡片包含：健康评分、关键异常、当日执行summary、趋势箭头
- 支持点击"详细报告"跳转到飞书文档或Web面板
- **优点**：主动推送、移动友好、低认知负荷
- **缺点**：信息量受限、需频繁查看聊天记录

**方案D：本地CLI命令（极客型）**
- `openclaw health` 命令，输出ASCII图表+关键指标
- 支持 `--detail` 查看子系统详情，`--fix` 尝试自动修复已知问题
- **优点**：快速、无网络依赖、开发者友好、可管道化
- **缺点**：需打开终端、学习曲线、不适合非技术展示

**方案E：系统托盘+桌面通知（沉浸型）**
- Windows系统托盘图标，颜色反映健康状态（绿/黄/红）
- 鼠标悬停显示summary，点击打开详细面板
- 异常时弹出桌面通知（可配置阈值）
- **优点**：零点击可见、即时感知、OS集成度高
- **缺点**：需常驻后台进程、多显示器支持复杂、移动端不可用

**方案F：Markdown静态页面（GitHub Pages型）**
- 生成`health-dashboard.md`，自动commit到workspace/.health/目录
- 通过GitHub Pages或本地文件浏览，使用简单JS图表
- **优点**：版本化历史、可离线查看、版本对比容易
- **缺点**：更新延迟、依赖Git操作、无实时推送

**补充创意（不限于以上）：**
- 集成到HEARTBEAT.md：每次heartbeat自动追加健康快照
- 语音助手播报：TTS每天早上播报"系统健康状况简报"
- WhatsApp/Telegram机器人：查询`/health`获取即时状态

### 阶段4：原型 - 选择方案快速设计MVP

**选择方案：B + C 组合（飞书文档表格 + 聊天卡片摘要）**

**选择理由（基于第二阶段定义的成功标准）：**
1. **零部署门槛**：无需额外服务器/服务，飞书权限已就绪（`docx:document:create/write_only` ✅）
2. **信息聚合能力**：表格可容纳多维度数据（时间序列、子系统状态、异常记录）
3. **移动友好**：主人可在手机查看，适合on-the-go场景
4. **自动化友好**：heartbeat cron每4h触发，可同步更新表格+发送卡片
5. **快速验证**：2小时内可建成可工作原型，明天即可收集反馈

   - 聊天卡片打开率（如果可追踪click-through）

   - 调整健康评分权重（如emotion-training权重从20%降至10%），观察主人满意度变化

**Iteration 1（第1周）：MVP验证**

**Iteration 2（第2-3周）：信息优化**
- 基于使用数据，合并冗余字段，突出关键异常
- 增加"建议操作"列（如"OpenRouter配额不足 → 建议充值或切换"）

- 实现异常根因分析（如cron失败 → 检查provider状态、网络延迟、文件锁）

- 健康评分与实际异常检测的准确率<70% → 算法需重构
- 维护成本超过收益（每次heartbeat cron延迟增加>5min）→ 简化或暂停
